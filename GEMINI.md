# Cookmark Project Instructions (KR/JP)

## 한국어 (Korean)

`Cookmark` 프로젝트를 위한 기본 규정입니다.

### 프로젝트 개요

`Cookmark`는 다양한 링크(YouTube, Instagram 등)에서 레시피를 저장하고 정리하기 위해 설계된 개인용 레시피 관리 도구입니다.

### 기술 스택

- **프론트엔드:** SvelteKit (Svelte 5) 및 Tailwind CSS 4.
- **백엔드:** Go (표준 라이브러리/Gin 스타일 구조).
- **데이터베이스:** PostgreSQL.
- **아키텍처:** `/frontend` 및 `/backend`를 포함하는 모노레포.

### 디자인 컨벤션

- **반응형 전략:** "플랜 A: 다중 열 대시보드".
  - **모바일:** 고정된 하단 네비게이션이 있는 단일 열 레이아웃.
  - **데스크톱 (lg):** 네비게이션/통계를 위한 고정된 왼쪽 사이드바와 그리드 기반의 메인 콘텐츠 영역이 있는 2열 레이아웃.
- **시각적 스타일:** 따뜻하고 유기적인 느낌.
  - 배경색: `#EDE9E2` (Warm Ivory).
  - 주요 강조색: `#7C9A7E` (Sage Green).
  - 콘텐츠 컨테이너: 은은한 그림자가 있는 흰색.

### 엔지니어링 표준

- **Svelte:** Svelte 5 룬(`$state`, `$props`, `$derived` 등)을 사용하세요.
- **Tailwind:** Tailwind 4 기능을 사용하세요. 필요한 경우가 아니면 커스텀 CSS보다 유틸리티 클래스를 우선하세요.
- **Go:** 관용적인(idiomatic) Go 패턴을 따르세요. 비즈니스 로직은 `internal/service`에 위치합니다.
- **Git:** `.env` 파일을 커밋하지 마세요. `.gitignore`가 준수되는지 확인하세요.

### 상업적 이용 안전성 (Commercial Safety)

- **라이선스 준수:** MIT, Apache 2.0 등 상업적 이용이 자유로운 라이선스의 라이브러리 및 코드를 우선적으로 사용하세요.
- **Copyleft 회피:** GPL 등 소스코드 공개 의무가 발생하는 코드를 프로젝트에 직접 복사하거나 포함하지 마세요.
- **보안 관행:** 상업적 서비스 수준의 보안(API 키 보호, SQL 인젝션 방어 등)을 엄격히 준수하세요.

### 워크플로우

- **개발:** 프론트엔드와 백엔드를 별도로 실행하거나 Docker Compose를 통해 실행하세요.
- **테스트:** 각 디렉토리에 새로운 기능에 대한 테스트를 추가하세요.
- **리팩토링:** 새로운 페이지를 추가할 때 "플랜 A" 반응형 구조를 유지하세요.

## 日本語 (Japanese)

`Cookmark` プロジェクトの基本規定です。

### プロジェクト概要

`Cookmark` は、様々なリンク（YouTube, Instagramなど）からレシピを保存し、整理するために設計された個人用レシピ管理ツールです。

### 技術スタック

- **フロントエンド:** SvelteKit (Svelte 5) と Tailwind CSS 4。
- **バックエンド:** Go (標準ライブラリ/Gin라이크な構造)。
- **データベース:** PostgreSQL。
- **アーキテクチャ:** `/frontend` と `/backend` を含むモノレポ。

### デザイン規約

- **レスポンシブ戦略:** 「プランA：マルチカラムダッシュボード」。
  - **モバイル:** 固定の下部ナビゲーションを備えたシングルカラムレイアウト。
  - **デスクトップ (lg):** ナビゲーション/統計用の固定左サイドバーとグリッドベースのメインコンテンツエリアを備えた2カラムレイアウト。
- **視覚的スタイル:** 温かみのあるオーガニックな雰囲気。
  - 背景色: `#EDE9E2` (Warm Ivory)。
  - 主要アクセント色: `#7C9A7E` (Sage Green)。
  - コンテンツコンテナ: 微かな影のある白。

### エンジニアリング標準

- **Svelte:** Svelte 5 の Runes (`$state`, `$props`, `$derived` など) を使用してください。
- **Tailwind:** Tailwind 4 の機能を使用してください。必要な場合を除き、カスタムCSSよりもユーティリティクラスを優先してください。
- **Go:** 慣用的な (idiomatic) Go パターンに従ってください。ビジネスロジックは `internal/service` に配置します。
- **Git:** `.env` ファイルをコミットしないでください。 `.gitignore` が守られていることを確認してください。

### 商用利用の安全性 (Commercial Safety)

- **ライセンス遵守:** MIT, Apache 2.0 などの商用利用が自由なライセンスのライブラリやコードを優先的に使用してください。
- **コピーレフトの回避:** GPL など、ソースコード公開義務が発生するコードをプロジェクトに直接コピーしたり含めたりしないでください。
- **セキュリティ慣行:** 商用サービスレベルのセキュリティ（APIキーの保護、SQLインジェクション対策など）を厳格に遵守してください。

### ワークフロー

- **開発:** フロントエンドとバックエンドを個別に実行するか、Docker Compose を介して実行してください。
- **テスト:** 各ディレクトリに新機能のテストを追加してください。
- **リファクタリング:** 新しいページを追加する際は、「プランA」のレスポンシブ構造を維持してください。
