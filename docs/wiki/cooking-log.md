# 쿠킹 로그 기능 컨텍스트

## 현재 구조 요약

- `cooking_logs` 테이블: 레시피 참조, 코멘트, 조리 일시
- RecipeNote를 대체함 (→ `docs/decisions/002-cooking-log-replaces-recipe-note.md`)

## 채택 안 한 접근법

### 재료 소비 삭제 — 프론트엔드 매칭
레시피 재료 ↔ 냉장고 목록 매칭을 클라이언트에서 하는 방향 검토 → 사용자가 백엔드 처리를 선호해서 `GET /api/recipes/{id}/pantry-matches` 신규 엔드포인트로 결정

### 재료 소비 삭제 — 별도 API 호출 (2-step)
쿠킹로그 저장 후 프론트에서 `DELETE /api/pantry/{id}` 개별 호출하는 방식 → 로그는 저장됐는데 재료는 안 지워지는 비원자적 상태가 생길 수 있어서 기각. `POST /api/recipes/{id}/logs` 하나로 트랜잭션 처리.

### 재료 선택 UI — 모달 이전 단계로 분리
"만들었어" 클릭 → 재료 선택 → 날짜/코멘트 입력 순서로 2단계 흐름 검토 → 기존 폼에 재료 선택 섹션을 합쳐서 모달 1개로 해결하는 게 단계를 늘리지 않아서 채택

## 발견한 제약사항·함정

### CreateAndDeletePantry의 cross-table 레포 패턴
`PgCookingLogRepository`가 `my_pantry` 테이블을 직접 DELETE 함. 레이어 분리 원칙상 이상하지만, 트랜잭션을 서비스 레이어에서 관리하려면 pool을 서비스에 노출해야 해서 실용적 선택으로 repo에서 처리. 소규모 프로젝트 한정 허용.

## 트레이드오프 & 맥락

### 만들었어 — 재료 소비 기능 설계 (2026-06)

냉장고에 있는 재료를 레시피 조리 후 매번 직접 삭제하러 가는 UX 개선 목적.

- **매칭 기준**: 냉장고 ↔ 레시피 재료 substring 매칭 (기존 추천 엔진과 동일 로직). FK 없이 이름 기반 매칭이므로 과매칭 가능성 있지만, 사용자가 체크박스로 최종 선택하므로 허용.
- **삭제 선택 optional**: 냉장고에 아무것도 없어도 "만들었어"는 눌러야 하므로, 재료 선택 없이도 저장 가능.
- **기본 전체 체크**: 보통 레시피대로 전부 만들었으면 전부 소비한 거라 전체 체크가 기본. 예외적으로 안 쓴 재료만 해제.
- **전체선택/전체해제**: 재료가 많을 때 편의성 위해 추가.
